## Summary

Removes Kafka's dependency on ZooKeeper by storing all cluster metadata in an internal Raft-based quorum run by the controllers themselves.

## Motivation

ZooKeeper is a second distributed system operators must deploy, secure, monitor and tune. It doubles the operational surface and forces a separate security and configuration model on top of Kafka's own.

Metadata scalability is bounded by ZooKeeper. On controller failover the new controller must load the full cluster state from ZooKeeper, and propagate it broker-by-broker — an O(partitions) operation that caps practical partition counts and lengthens recovery.

## Proposed Changes / Design

Metadata is stored in a single-partition internal topic, __cluster_metadata, replicated by a Raft quorum. The active controller is the quorum leader; other controllers are hot standbys that already have the full log.

Brokers stop watching ZooKeeper and instead consume the metadata log as an ordered event stream, caching state locally. Failover becomes a matter of a standby taking over leadership with the log already in memory — near-instant rather than an O(partitions) reload.

## Trade-offs

> [!success]+ Benefits
> - One system to deploy, secure and operate instead of two.
> - Controller failover is near-instant; recovery no longer scales with partition count.
> - Supports clusters with millions of partitions.
> - Single, consistent security and RBAC model.

> [!warning]+ Costs / Risks
> - A brand-new consensus implementation to harden and battle-test.
> - Multi-release migration path for existing ZooKeeper clusters.
> - Operators must relearn tooling, metrics and failure modes.

## Rejected Alternatives

**Keep ZooKeeper but sharded** — Sharding metadata across multiple ZK ensembles would reduce Kafka's control over consistency and add even more operational moving parts.

**Use an external Raft library** — An off-the-shelf library couldn't be tuned to Kafka's log semantics and record format; a purpose-built quorum integrates with the existing replication machinery.
