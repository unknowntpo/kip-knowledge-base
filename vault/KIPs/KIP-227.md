## Summary

Replaces full fetch requests with a stateful, incremental session so brokers and consumers only exchange data for partitions that actually changed.

## Motivation

Every fetch request listed every partition and its offset, even for idle partitions. With thousands of partitions the request/response overhead dominated, wasting network and CPU.

This overhead scaled with total partition count rather than active partitions, capping how many partitions a broker or consumer could follow efficiently.

## Proposed Changes / Design

A fetch session is established once; the broker tracks the client's per-partition state under a session ID. Subsequent incremental fetches send only partitions that changed since the last round.

Idle partitions are omitted entirely, so steady-state fetch traffic scales with the number of active partitions, not the total.

## Trade-offs

> [!success]+ Benefits
> - Fetch overhead scales with active, not total, partitions.
> - Substantially higher partition density per broker and consumer.
> - Less network and CPU for large, mostly-idle topics.

> [!warning]+ Costs / Risks
> - Brokers must maintain per-session state and evict stale sessions.
> - More complex fetch protocol and error/recovery handling.

## Rejected Alternatives

**Compress the full fetch request** — Compression shrinks bytes but not the per-partition bookkeeping and CPU cost; it doesn't change the O(total partitions) scaling.
